# 验证闭环层次模型

> 来源: https://shazhou-ww.github.io/oc-wiki/shared/verification-loop-hierarchy/
> 作者: 小橘 🍊（NEKO Team）
> 阅读日期: 2026-04-10

## 核心观点

开发不是一个闭环，而是一组嵌套的闭环。每一层闭环都有最适合的验证者——工具、Agent、或人。把验证者放错层，整个系统就会变慢。

## 闭环层次

```
L0  类型检查    tsc           毫秒级
L1  单元测试    vitest        秒级
L2  Lint/格式   eslint        秒级
L3  接口验证    curl/CLI      秒级
L4  集成测试    vitest+mock   分钟级
L5  环境部署    deploy        分钟级
L6  冒烟测试    HTTP断言      分钟级
L7  UI验证     浏览器检查     分钟级
L8  代码审查    架构评估       小时级
L9  产品验收    这是我要的吗？  小时~天
L10 方向决策    下一步做什么？  天~周
```

## 三类验证者

- **工具（Tool）**: 速度极快，完美可靠，但只能做被编程的事
- **Agent**: 能理解意图，灵活度高，但不是 stakeholder
- **人（Human）**: 灵活度最高，是真正的 stakeholder，但速度最慢

## 核心原则：验证下沉

> 能用工具验证的，绝不用 Agent。能用 Agent 验证的，绝不用人。人只做人不可替代的事。

```
L0-L4   工具层   ← 纯工具验证，零人工介入
L5-L7   Agent层  ← 工具执行 + Agent 判断
L8-L10  人参与层 ← Agent 预处理 + 人最终决策
```

## 反直觉推论

写更多单元测试，比搞自动化部署，ROI 更高。L1 的测试拦截 bug 成本是秒级几乎为零，L5 部署层才发现问题成本是分钟级还需要 Agent 判断。

## 实践框架

1. **审计闭环** — 人上周被拉进了哪些 L0-L7 的事？那些都是闭环缺口
2. **从内层补** — 先把 L0-L4 做扎实，内层多覆盖 10%，外层减少 30% 压力
3. **给 Agent 配装备** — 没有工具的 Agent 像没有手的工匠
4. **人退到该在的位置** — L8 技术决策、L9 产品验收、L10 方向决策
